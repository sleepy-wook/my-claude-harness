# core-rules

## 표준 작업 합의 (standing working agreements)

- 이 개발자는 명시적으로 요청하지 않은 변경(요청 범위를 벗어난 새 파일 대량 생성, 비요청 리팩터링, 임의의 스코프 확장)을 선호하지 않는다.
- 이 개발자는 "테스트·검증을 실제로 실행하지 않은 채 '완료됐다'고 말하는 것"을 신뢰하지 않는다. 완료 주장은 실제 실행 결과(테스트 통과 출력, 명령 exit 0 등)로 뒷받침되어야 한다.
- 이 개발자는 추측으로 진행하기보다, 불확실하면 멈추고 확인받는 방식을 선호한다.

## 하네스 워크플로 (기본값)

- 이 개발자는 중간 규모 이상의 구현을 `/wook-plan`으로 시작해 수용 기준을 먼저 정하는 것을 기본 흐름으로 삼는다(그 기준이 `.claude/evaluate.recipe`가 되고, 자동 게이트가 완료를 그 기준으로 검증한다). 새 기능/모듈 구현 요청인데 `.claude/plan.md`가 없으면 코드 작성 전 `/wook-plan`을 먼저 제안한다(단순 수정/탐색/질문은 제외).
- 이 개발자는 새 함수·컴포넌트·쿼리를 만들기 전에, 프로젝트에 재사용 카탈로그(`.claude/reuse-index/`)가 있으면 거기서 기존 것을 먼저 재사용하는 흐름을 선호한다.
- 이 개발자는 재사용할 만한 함수·컴포넌트·쿼리를 새로 만들면, 카탈로그가 있는 프로젝트에서는 해당 도메인 `.claude/reuse-index/<domain>.md`에 한 줄(`이름 · 한줄설명 · path:symbol`)을 추가해 카탈로그를 최신으로 유지하는 흐름을 선호한다.
- 이 개발자는 프로젝트에 코딩 컨벤션(`.claude/conventions/`)이 있으면 해당 도메인 작업 시 `<domain>.md`(공통 규칙은 `shared.md`)를 따르며, 컨벤션을 새로 정하거나 바꾸거나 폐기하면 같은 작업 안에서 그 문서를 갱신해 최신으로 유지하는 흐름을 선호한다.
