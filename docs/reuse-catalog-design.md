# 재사용 카탈로그 — 설계 스펙 (reuse catalog)

> 2026-06-09 브레인스토밍 결과. **구현 전 합의된 설계.** 승인되면 build-log/구현으로.

## 문제 / 목표
AI가 프로젝트의 기존 컴포넌트·함수·쿼리를 **전체 코드를 안 읽고 짧은 인덱스만 보고 재사용**
(중복 생성 방지). 도메인(프론트/백/DB/…) 무관하되, **도메인별로 분리**해서 필요한 도메인 것만.

## 핵심 설계 — per-project, 2단계(retrieval) 카탈로그
Snowflake/Databricks 패턴: **설명 목록 → 관련 것만 선택 → 그 상세만 사용.** Skill.md 형식엔
묶이지 않음(자유·전체 제어).

- **Tier-1 매니페스트 (도메인별):** `.claude/reuse-index/<domain>.md`
  - 도메인 = **파일 있는 만큼**(frontend/backend/db…, 고정 아님, 프로젝트마다 자유).
  - 항목 형식: `이름 · 한줄설명 · 위치(path:symbol)`.
- **Tier-2 상세:** 포인터가 가리키는 **실제 소스를 그 자리에서 Read** → 상세는 **절대 안 낡음**.
  매니페스트(한줄설명·포인터)만 가끔 갱신하면 됨.

## 컴포넌트
1. **매니페스트 파일** — md, 도메인별. (`.claude/reuse-index/<domain>.md`)
2. **포인터 주입 hook** (UserPromptSubmit) — `.claude/reuse-index/`가 있으면 매 턴 **가벼운 안내만**
   주입: *"재사용 매니페스트가 도메인별로 있음: [frontend, backend, db]. 그 도메인 새 코드 짜기
   전 해당 파일 확인."* (매니페스트 본문 아님 = 도메인 라우팅 인지만). 기존 `inject_core_rules`
   패턴 확장/별도 hook.
3. **`/wook-index` 생성 스킬** (전역, 하네스에 추가) — 코드 훑어 도메인 분류 + 도메인별 매니페스트
   생성/갱신. AI가 이해(멀티 stack 무관). 재사용물 추가 후 재실행으로 최신화.

## 동작 흐름
1. (매 턴) 포인터 주입 → AI가 "어느 도메인에 재사용 매니페스트가 있는지" 항상 인지.
2. AI가 **작업 도메인의 매니페스트 1개만** Read(프론트=frontend.md만) → 후보 선택 → **실제 소스
   Read** → 재사용(import/호출). (프론트 짤 때 백/DB 데이터 안 들어옴.)
3. 새 재사용물 추가 → `/wook-index` 재실행 → 매니페스트 갱신.

## 활성화 / 안전 / 비용
- `.claude/reuse-index/` 존재 = 활성(없으면 포인터 주입 0). 게이트와 같은 "파일 존재=ON" 패턴.
- 주입은 **가벼운 포인터만**(상한 둠). 매니페스트 본문·상세는 온디맨드 → 컨텍스트 절약.
- 매니페스트는 commit(팀 공유 가능). 상세=실제 소스라 스테일 없음; 포인터만 재생성 대상.

## 수용 기준 (구현 시 게이트로 검증)
- `/wook-index` 실행 → `.claude/reuse-index/<domain>.md` 생성, 각 항목에 `path:symbol` 포인터 포함.
- 포인터 hook: reuse-index 있는 프로젝트에서 도메인 목록 주입(파이프 테스트); 없으면 무출력 exit 0.
- **스테일 포인터 없음**: 매니페스트의 모든 path가 실제 존재(selfcheck류 스크립트로 검증).

## 향후 / 관련 (이 스펙 밖, 별도 논의)
- 포인터에 도메인 이름까지 넣기(추천: 넣음 — 라우팅 도움).
- `/wook-index` 자동 최신화(코드 많이 바뀌면 stale 경고) — 나중.
- **standing 지침 최적화(별도 작업):** 이 워크플로(+`/wook-plan`·게이트·재사용)를 기본값으로.
  → core-rules(매 턴 주입, 강함, 짧게) vs 전역 `~/.claude/CLAUDE.md`(1회 로드) 배치 결정.
  **카탈로그 구현 후** 진행(완성된 컴포넌트를 정확히 참조하도록).
